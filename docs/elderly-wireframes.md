# 高齢者向け画面 ワイヤーフレーム確定版

## 1. 基本方針

高齢者向け UI は `1画面1目的` を徹底する。

- 1つの画面で複数判断をさせない
- 数字や専門用語を見せすぎない
- 文字を読むより、短い説明と大きな選択肢で進める
- ロボット体験が主役なので、UIは補助に徹する

## 2. MVP で作る画面

今回の高齢者向け画面は 3 つに固定する。

1. 散歩前 `今日のお散歩提案`
2. 散歩中 `いまのお散歩`
3. 散歩後 `今日の感想`

家族向け画面、設定画面、履歴画面はこの段階では含めない。

## 3. 画面遷移

```text
[今日のお散歩提案]
   ├─ このコースでいく → [いまのお散歩]
   ├─ 5分の短いコース → [いまのお散歩]
   └─ 今日はやめる → 終了

[いまのお散歩]
   └─ 散歩終了 → [今日の感想]

[今日の感想]
   └─ 送信 → 終了
```

## 4. 画面 1: 今日のお散歩提案

### 目的

今日、散歩に行くかどうかを決める。

### 必要情報

- 今日の提案時間
- コースの説明
- 提案理由
- 選択肢

### ワイヤーフレーム

```text
+----------------------------------+
| きょうのおさんぽ                  |
|----------------------------------|
| きょうは 8分 のおさんぽです       |
| いつもの あんしんな道です         |
|                                  |
| [ やさしい地図イメージ ]           |
|                                  |
| りゆう                           |
| ・すこし暑いので短めです          |
| ・日かげの多い道です              |
|                                  |
| [ このコースでいく ]              |
| [ 5分のコースにする ]             |
| [ 今日はやめる ]                  |
+----------------------------------+
```

### UI ルール

- 主ボタンは 1 つだけ強く見せる
- 代替案は 1 つまで
- 難しい指標は出さない
- 理由は 2 行まで

### API 接続

- 画面表示時に `POST /api/v1/walk-routes/suggest`
- `このコースでいく` で散歩開始
- `5分のコースにする` は短い代替案を選ぶ

### 表示文言ルール

- `WBGT 24.1` ではなく `すこし暑い`
- `推奨経路` ではなく `いつもの道`
- `代替ルート` ではなく `短いコース`

## 5. 画面 2: いまのお散歩

### 目的

散歩中に、今どういう状態かを安心して把握する。

操作はスマホではなく、将来的にリードによるインタラクションで行う前提とする。

### 必要情報

- 散歩中であること
- 残り時間の目安
- 今の状態

### ワイヤーフレーム

```text
+----------------------------------+
| いまのおさんぽ                    |
|----------------------------------|
| のこり 4分くらい                  |
|                                  |
| いまは ゆっくり歩いています       |
|                                  |
| [ シンプルな進行表示 ]            |
+----------------------------------+
```

### UI ルール

- 地図は必須ではない
- 現在地の精密表示はしない
- 「今どういう状態か」だけ分かればよい
- 緊急時以外は警告色を多用しない
- 散歩中の操作ボタンは置かない

### 状態バリエーション

#### 通常

- `いまは ゆっくり歩いています`

#### 休憩中

- `いまは きゅうけいしています`

#### 帰宅提案時

- `そろそろ帰りましょう`

### API 接続

- 散歩開始時に `POST /api/v1/walk-sessions`
- 散歩中イベントを `POST /api/v1/walk-sessions/:walkSessionId/events`
- 散歩終了はロボット側イベントまたはリード操作をトリガに `POST /api/v1/walk-history`

### リード連携前提

- `止まる`
- `休憩する`
- `帰る`
- `終了する`

これらの操作はスマホUIではなく、将来的にリード入力で解釈する。

散歩中画面は「操作する画面」ではなく、「いまの状態を確認する画面」として扱う。

## 6. 画面 3: 今日の感想

### 目的

散歩後の疲労度と満足度を、迷わず記録する。

### 必要情報

- 疲れたか
- 楽しかったか
- この道はよかったか
- 送信ボタン

### ワイヤーフレーム

```text
+----------------------------------+
| おさんぽどうでしたか              |
|----------------------------------|
| つかれましたか                    |
| [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]    |
|                                  |
| たのしかったですか                |
| [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]    |
|                                  |
| この道はどうでしたか              |
| [ よかった ] [ ふつう ] [ いまいち ]|
|                                  |
| [ 送信する ]                      |
+----------------------------------+
```

### UI ルール

- 入力項目は 3 つまで
- 自由記述は初期MVPでは入れない
- 送信ボタンは画面下に固定

### API 接続

- `POST /api/v1/walk-history`
- 必要なら後から `POST /api/v1/walk-history/:walkHistoryId/feedback`

## 7. 画面別コンポーネント

## 7.1 共通

- ページタイトル
- 短い説明文
- 大きい主ボタン

## 7.2 今日のお散歩提案

- `WalkSummaryCard`
- `ReasonList`
- `PrimaryActionButton`
- `SecondaryActionButton`
- `CancelButton`

## 7.3 いまのお散歩

- `WalkStatusCard`
- `ProgressIndicator`
- `WalkStateLabel`
- `RemainingTimeLabel`

## 7.4 今日の感想

- `FatigueSelector`
- `EnjoymentSelector`
- `RouteFeedbackSelector`
- `SubmitFeedbackButton`

## 8. デザイン制約

- ボタンは縦並び優先
- タップ領域は広く取る
- 1画面の選択肢は最大 3 個
- 色で意味を持たせすぎない
- 文字サイズは大きめを前提にする

## 9. 実装時の注意

- 本人向け画面では UUID や routeId を見せない
- 地図は詳細地図でなく、簡易イメージでよい
- 散歩中画面は「見守り」ではなく「安心感」を優先する
- 散歩中の操作はUIでなくリード側に寄せる
- 疲労入力は将来の `load_score` 更新に直接つながる

## 10. 今回の確定事項

高齢者向けMVP画面は以下で確定。

1. `今日のお散歩提案`
2. `いまのお散歩`
3. `今日の感想`

これ以外の画面は、初期フロントでは作らない。
